# Phase 5: 通用数据验证与修复机制

## 设计目标
Phase 5的目标是确保生成的JSON数据具有通用的有效性和一致性，而不是针对特定文档结构进行验证。

## 通用验证规则

### 1. JSON结构验证
- 验证JSON格式是否合法
- 验证根元素是否为数组
- 验证数组中的每个元素是否为对象

### 2. 必需字段验证
- 每个对象必须包含以下字段：
  - `type`: 项目类型 ("context" 或 "question")
  - `number`: 项目编号
  - `content`: 项目内容

### 3. 类型验证
- `type` 字段必须是字符串，且值为 "context" 或 "question"
- `number` 字段必须是字符串
- `content` 字段必须是字符串
- 对于 `type` 为 "question" 的对象，还必须包含 `answer` 和 `analysis` 字段

### 4. 内容验证
- `content` 字段不能为空字符串或仅包含空白字符
- `type` 字段值必须合法（"context" 或 "question"）

### 5. 顺序验证
- 按照 `number` 字段进行排序
- 验证数字编号是否连续（仅适用于纯数字编号）
- 保持 `context` 和 `question` 的逻辑顺序

## 修复与构建机制

### 1. 通用数据修复工具 (`utils/data_repair.py`)
- **功能整合**: 将原有 `repair/` 目录中的功能整合至此
- **JSON数据序列完整性验证**: 检测题目编号是否连续
- **自动修复缺失题目**: 从备份源获取缺失内容
- **数据顺序重组**: 按逻辑关系重新排列context-question对
- **错误数据处理与恢复**: 从损坏JSON中提取有效数据
- **综合数据校验**: 全面验证数据完整性

### 2. 数据验证工具 (`utils/data_validator.py`)
- **功能整合**: 将原有 `validation/` 目录中的功能整合至此
- **JSON结构验证**: 验证数据格式和必需字段
- **序列连续性验证**: 检查编号连续性
- **字段完整性验证**: 检查必需字段的存在性

### 3. 构建工具 (`utils/build_tools.py`)
- **功能整合**: 将原有 `build/` 目录中的功能整合至此
- **从分块构建**: 从chunks文件构建extracted数据
- **数据重排序**: 确保context和question的逻辑顺序
- **后处理验证**: 构建后的数据完整性验证

### 4. 字段补全
- 对于缺少必需字段的对象，使用默认值补全
- 对于 `question` 类型，为 `answer` 和 `analysis` 字段提供默认值

### 5. 非法字符清理
- 清理JSON字符串中的非法转义字符
- 修正可能导致解析错误的格式问题

### 6. 结构修复
- 将嵌套的数组结构展平
- 修正对象结构中的错误

## 重要限制

### 禁止特定内容验证
以下验证规则**绝对禁止**使用：
- 硬编码的编号段验证（如"一"必须对应"1-3"题）
- 特定文档结构验证（如"一到七"大题的数量验证）
- 固定的题号范围验证（如特定题号范围必须存在）
- 针对特定文档内容的逻辑验证

### 通用性原则
- 所有验证规则必须适用于任何类型的文档
- 不得假设文档具有特定结构或内容
- 验证逻辑应关注数据完整性而非内容特定性
