# 项目开发文档：智能试卷转 PPT 助手 (Exam2PPT)

## 1. 项目概述

本项目旨在开发一个 Windows 桌面应用程序 (`.exe`)，利用 LLM (DeepSeek API) 将 Word 格式的试卷文档（`.docx`）自动转换为教学用的 PowerPoint 演示文稿（`.pptx`）。

**核心价值**：自动化完成“试卷阅读 -> 结构化提取 -> PPT排版”的繁琐过程，无需本地显卡，低配置机器即可运行。

## 2. 技术栈选型

* **开发语言**：Python 3.10+
* **GUI 框架**：`Tkinter` (Python 内置，轻量级) 或 `CustomTkinter` (更现代美观，推荐)。
* **文档处理**：
* 读取：`python-docx`
* 写入：`python-pptx`


* **AI 交互**：`openai` SDK (用于调用 DeepSeek API，兼容 OpenAI 格式)。
* **打包工具**：`PyInstaller`
* **并发控制**：`threading` (用于界面防卡死和终止操作)。

## 3. 功能需求 (FR)

### 3.1 用户界面 (GUI) 设计

界面应简洁直观，包含以下模块：

1. **API 配置区**：
* 输入框：输入 DeepSeek API Key (带掩码显示 `****`)。
* 输入框：API Base URL (默认为 `https://api.deepseek.com`, 允许修改)。
* **[新增] 模型选择**：下拉菜单支持选择 `DeepSeek-V3` (Chat) 或 `DeepSeek-R1` (Reasoner)。


2. **文件操作区**：
* **[按钮] 选择试卷文档**：弹出文件选择框，仅限 `.docx`。显示选中的文件路径。
* **[按钮] 选择保存路径**：弹出文件夹选择框。默认为源文件所在目录。


3. **控制区**：
* **[按钮] 开始转换**：点击后禁用，开始执行任务。
* **[按钮] 终止任务**：点击后立即停止当前逻辑，重置界面状态。


4. **状态反馈区**：
* **进度条**：显示当前处理进度（读取 -> AI 处理 -> 生成 PPT）。
* **日志窗口/文本框**：实时打印当前正在处理的步骤（如“正在分析第 1-10 题...”）。



### 3.2 核心业务逻辑 (The "Brain")

#### 3.2.1 文本读取与切片 (Anti-Data-Loss Logic)

* **需求**：支持 5 万字以内的文档解析。
* **痛点解决**：LLM 输出长度有限，一次性输入 5 万字会导致 JSON 截断；简单切分会导致题目截断。
* **算法逻辑 (Updated)**：
1. **预扫描**：使用正则表达式 (如 `^\d+\.|^Question \d+|^\(\d+\)`) 扫描全文，识别所有题号位置。
2. **结构化切分**：基于题号位置将文本分割为独立的“题目块”。
3. **打包发送**：将多个题目块合并为一个 Chunk（约 2000-3000 tokens），确保每个 Chunk 包含完整的 N 道题。
* **优势**：彻底解决“半道题”导致的解析失败问题。





#### 3.2.2 AI 结构化处理

* 调用 DeepSeek `chat/completions` 接口。
* **System Prompt (Updated)**:
> 你是一个严格的数据提取器，不是摘要助手。
> 禁止总结：解析字段中的 `context` 和 `analysis` 必须完全保留原文，**一个字都不能删减**。
> 完整性检查：如果你收到的文本包含 N 道题，你的 JSON 数组必须包含 N 个对象，**绝不能跳过任何一道题**。
> 如果某道题太长，请完整保留，不要用 "..." 代替。
> 
> 格式要求：
> 1. 返回一个列表，包含多个题目对象。
> ... (同上)2. 每个对象包含字段：`id` (题号), `type` (题型), `context` (题干/大段阅读材料，如果没有则为空), `question` (具体问题文本), `options` (选项数组，如["A. xxx", "B. xxx"]), `answer` (答案), `analysis` (解析)。
> 3. 不要输出任何 Markdown 标记（如 ```json），直接输出 JSON 字符串。
> 4. 必须确保 JSON 格式合法。
> 
> 


#### 3.2.3 PPT 生成逻辑

* **模板机制**：程序内部应内置一个基础 PPT 模板（包含母版布局），防止生成的 PPT 白底黑字太丑。
* **排版规则**：
* **封面页**：显示文件名。
* **题目页**：
* 如果有 `context` (背景材料)，单独占一页或置于顶部。
* 显示 `question` 和 `options`。


* **答案页**（紧跟题目页后，或作为动画出现）：
* 显示 `answer` 和 `analysis`。
* 字体颜色建议：答案用红色或高亮显示。






#### 3.2.5 完整性校验 (Watchdog)

* **双重校验机制**：
1. **输入端计数**：正则统计 Word 中识别到的题目数量（Expected Count）。
2. **输出端计数**：统计所有 AI 返回的 JSON 列表中的 `id` 总数（Actual Count）。
3. **触发警报**：如果 Expected != Actual，在日志中记录 WARNING，并在 GUI 弹窗提示用户：“检测到可能存在漏题现象（原文档约 X 题，生成了 Y 题）”。

#### 3.2.6 PPT 文本溢出处理

* **分页机制**：生成 PPT 时检测文本长度。
* 如果解析（Analysis）或题干超过一定阈值（如 300 字），自动拆分为多页 PPT 展示（Page 1: 题目+选项; Page 2: 答案+解析）。

### 3.3 非功能需求 (NFR)

#### 3.3.1 日志系统 (Logging)

* 使用 Python `logging` 模块。
* **保存位置**：程序同级目录下的 `app.log`。
* **记录内容**：
* [INFO] 程序启动。
* [INFO] 开始处理文件：Filename.docx。
* [INFO] 发送第 N 个切片给 AI...
* [ERROR] API 连接失败：Error message。
* [ERROR] JSON 解析失败，原始返回内容：... (便于排查 AI 是否幻觉)。
* [INFO] 任务完成/终止。


* **GUI 联动**：日志不仅写入文件，也要同步打印到 GUI 的文本框中，方便用户直观看到报错。

#### 3.3.2 异常处理

* **网络错误**：DeepSeek API 偶尔超时，需增加 `retry` (重试) 机制，最多重试 3 次。
* **JSON 解析错误**：如果 AI 返回的不是标准 JSON，尝试使用 `json_repair` 库进行修复，或者记录错误跳过该片段，防止整个程序崩溃。

## 5. 补充开发建议 (给 AI 助手)

1. **关于 PPT 模板**：为了简化开发，第一版可以使用 `python-pptx` 创建一个空白 PPT，但要在代码中定义好 `Title Slide` 和 `Content Slide` 的布局索引。
2. **关于 Token 计算**：Word 文档可能包含图片，`python-docx` 读取时会自动忽略图片。请在代码注释中说明“当前版本暂不支持试卷中的图片提取”。
3. **关于流式输出**：由于需要生成 JSON，**不要**使用 `stream=True`，使用全量返回模式更易于解析。

## 6. 版本更新记录 (Changelog)

### v1.0 (2026-02-15) - 初始版本
* 实现了基础的 Word 读取、DeepSeek-V3 调用和 PPT 生成。
* 包含基础的 GUI 界面和日志系统。

### v1.1 - 模型选择支持
* **新增**：支持选择 DeepSeek-V3 (Chat) 或 DeepSeek-R1 (Reasoner) 模型。
* **修改**：GUI 界面新增模型下拉菜单。

### v1.2 - PPT 排版优化
* **优化**：调整了 PPT 字体大小（标题 28pt，正文 22pt/20pt）。
* **新增**：Context (背景材料) 分离为独立幻灯片，支持超长文本自动分页。
* **修复**：解决了文字溢出和排版拥挤的问题。

### v1.4 - 防丢数据安全机制 (Data Integrity Assurance)
* **新增**：核心补充模块：防丢字与完整性校验机制。
* **策略一**：基于结构的“智能切片” (Structural Chunking)。使用正则预扫描题号，按完整题目切分，杜绝“半道题”。
* **策略二**：强制“原文提取”模式 (Verbatim Extraction Prompt)。增强 Prompt 约束，禁止 AI 总结或删减内容。
* **策略三**：双重校验“看门狗” (The Watchdog Logic)。比对 Word 识别题数与 JSON 返回题数，异常时弹窗预警。
* **策略四**：PPT 文本溢出处理 (Visual Overflow Handling)。自动分业处理超长解析，防止文字溢出屏幕。
