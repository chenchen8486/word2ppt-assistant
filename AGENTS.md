# 智能体行为准则

## 核心原则

### 1. 极简工程主义
- 优先考虑简单直接的解决方案，避免过度工程化
- 当面临多个实现方案时，选择最简单的那个
- 保持代码库的简洁性和可读性
- 避免创建不必要的抽象层或复杂的目录结构

### 2. 功能完整性保障
- 保证核心业务功能的完整性，不得因重构而丢失功能
- 所有重构工作应遵循"保持功能不变，优化组织结构"的原则
- 每个功能模块应当职责明确且相互独立

### 3. 数据驱动的工作流
- 遵循解析(MarkItDown) -> 结构化(LLM+JSON) -> 渲染(python-pptx 模板驱动)的稳定流水线
- 确保数据在各个处理阶段的完整性和一致性
- 对数据流进行充分的验证和错误处理

### 4. 渐进式改进
- 逐步进行优化，避免大规模破坏性变更
- 每次改进都应该带来明确的价值
- 在每次重构后验证系统功能的完整性

## 行为指导

### 代码组织
- 按照功能逻辑将相关模块组织在一起
- 保持模块间的松耦合
- 使用清晰明了的命名约定
- 避免深层级的目录嵌套

### 错误处理
- 采取防御性编程策略
- 产生有意义的错误消息，而不是静默失败
- 保留足够的调试信息以便追踪问题

### 测试与验证
- 确保所有功能都有相应的验证机制
- 在集成测试中验证端到端的功能
- 维护数据完整性检查

## 项目演进策略

### 重构原则
- 重构的目的是提升可维护性，而不是功能扩展
- 在删除旧代码前确保新代码完全替代其功能
- 文档同步更新，保持与代码的一致性

### 架构演进
- 优先考虑架构的稳定性与可理解性
- 避免不必要的技术复杂度
- 保持系统的透明度，让任何开发者都能快速理解系统架构
