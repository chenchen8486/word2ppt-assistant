# Agent 开发工程标准

## 核心原则摘要

### 测试支架（Harness）
1. **长期运行代理的上下文桥接**：由于每个新会话开始时没有之前会话的记忆，必须建立机制让代理在多个上下文窗口之间保持连续性。
2. **初始化器代理模式**：使用专门的初始化代理设置初始环境，包括脚本、进度日志文件和初始提交，为后续编码代理提供必要上下文。
3. **代理可读性目标**：确保系统设计让代理能够理解代码库知识，并将其作为系统记录。

### 环境反馈循环
1. **身份持久性**：代理必须能够在时间上携带自己的身份，使其成为系统而非单一进程。
2. **状态外部化**：状态存在于单个上下文窗口之外，交互塑造代理行为而不仅仅是缓冲区。
3. **人类-AI反馈闭环**：AI操作应在一个封闭的人类-AI反馈环中进行。
4. **迭代改进循环**：驱动PR完成的流程包括本地审查更改、请求特定的代理审查、响应反馈并迭代直到所有代理审查者满意。

### 工程化质量把控
1. **增量进展机制**：编码代理应在每次会话中做出增量进展，同时为下次会话留下清晰的工件。
2. **存储库知识作为记录系统**：使存储库知识成为系统的记录，确保所有变更都有明确记录。
3. **计划与进度日志**：对于复杂工作，使用检查到存储库中的执行计划和进度决策日志。
4. **代理生成代码的质量验证**：建立机制验证代理生成代码的质量。

## 具体开发指令

### 解析逻辑实现前的准备
- 在实现任何解析逻辑前，必须先建立对应的测试用例
- 设计代码时必须包含自检功能，确保能够验证自身输出的正确性
- 实现状态持久化机制，使代理能够了解之前的执行历史

### 质量保证
- 所有代码必须具有可测试性，便于自动化验证
- 实现进度跟踪机制，在处理长时间任务时记录中间状态
- 设计错误恢复机制，当代理遇到问题时能够回滚或重试

### 环境管理
- 创建初始化脚本以设置代理工作的标准环境
- 建立标准化的日志记录机制，用于跟踪代理操作
- 实现上下文传递机制，确保多轮对话间的信息连续性

### 代码结构
- 遵循代理可读性原则设计代码架构
- 使用清晰的注释和文档，帮助代理理解代码意图
- 将复杂任务分解为可管理的小步骤，并为每一步提供明确的完成标准

### 测试策略
- 实施多层次测试策略（单元测试、集成测试、端到端测试）
- 为代理操作创建仿真环境，避免在生产环境中进行试验
- 建立回归测试套件，确保新功能不会破坏现有功能
